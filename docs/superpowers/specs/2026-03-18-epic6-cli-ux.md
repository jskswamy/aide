## Overview

This epic covers all user-facing CLI UX commands beyond the core `aide` launch flow. These commands help users understand what aide is doing (which, verbose), ensure correctness (validate), bootstrap configuration (init, setup), explore state (contexts, agents), integrate with their shell (completions), and get oriented on first use (first-run guidance).

All commands use cobra subcommands registered on the root command in `cmd/aide/main.go`.

## Task 23: `aide which` — Context Introspection

**File:** `cmd/aide/which.go`
**Internal packages used:** `internal/config`, `internal/context`

### Purpose

Show the fully resolved context for the current working directory, including which match rule triggered, and what other contexts could have matched. Users run this to answer: "If I ran `aide` right now, what would happen?"

### Output Format

```
Context:        personal
Match rule:     remote "github.com/jskswamy/*" matched origin "github.com/jskswamy/aide"
Agent:          claude (binary: /usr/local/bin/claude)
Secret:         personal
MCP servers:    git, context7, serena, things
Sandbox:        default policy (writable: project root, runtime dir)

Env vars (names only):
  ANTHROPIC_API_KEY  (from secrets)

Also matched (lower specificity):
  oss — remote "github.com/*" (less specific glob)

Project override: none
```

When a `.aide.yaml` override is present:

```
Context:        personal
Match rule:     remote "github.com/jskswamy/*" matched origin "github.com/jskswamy/aide"
Agent:          gemini (overridden by .aide.yaml)
...
Project override: /Users/you/project/.aide.yaml
  agent: gemini
  mcp_servers: [git, custom-server]
```

When in zero-config passthrough mode (no config.yaml):

```
Mode:           zero-config passthrough
Agent:          claude (detected on PATH: /usr/local/bin/claude)
Config:         none (~/.config/aide/config.yaml not found)
Hint:           Run `aide setup` to create a config, or just run `aide` to launch claude.
```

### Flags

| Flag | Description |
|------|-------------|
| `--json` | Output as JSON for scripting |
| `--context <name>` | Show what would resolve if this context were forced |

### JSON Output Schema

```json
{
  "mode": "configured",
  "context": "personal",
  "match_rule": {
    "type": "remote",
    "pattern": "github.com/jskswamy/*",
    "matched_value": "github.com/jskswamy/aide"
  },
  "agent": {
    "name": "claude",
    "binary": "/usr/local/bin/claude"
  },
  "secret": "personal",
  "mcp_servers": ["git", "context7", "serena", "things"],
  "sandbox": {
    "policy": "default",
    "writable": ["{{ .project_root }}", "{{ .runtime_dir }}"],
    "network": "outbound"
  },
  "env_var_names": ["ANTHROPIC_API_KEY"],
  "alternatives": [
    {
      "context": "oss",
      "match_rule": {"type": "remote", "pattern": "github.com/*"},
      "reason_not_selected": "less specific glob"
    }
  ],
  "project_override": null
}
```

### Implementation

1. Load config via `config.Load()`.
2. Run the context resolver (`context.Resolve()`) but capture ALL matches, not just the winner. The resolver must expose a `ResolveAll()` method that returns a ranked list of `MatchResult` structs.
3. Detect git remote via `context.DetectGitRemote()`.
4. Look up agent binary on PATH via `exec.LookPath()`.
5. If `--json`, marshal to JSON and print. Otherwise, format as human-readable text.
6. Exit 0 (this is informational, never fails unless config is broken -- in which case direct the user to `aide validate`).

### Required Types

```go
// internal/context/resolver.go

// MatchResult represents a context that matched the current environment.
type MatchResult struct {
    ContextName string
    MatchRule   MatchRule
    Specificity int // higher = more specific
}

// MatchRule describes which rule triggered the match.
type MatchRule struct {
    Type         string // "remote", "path", "project_override"
    Pattern      string // the glob pattern from config
    MatchedValue string // the actual git remote or path that matched
}

// ResolveAll returns all matching contexts, ordered by specificity (highest first).
func (r *Resolver) ResolveAll(gitRemote, cwd string) ([]MatchResult, error)
```

## Task 24: `aide validate` — Config Validation

**File:** `cmd/aide/validate.go`
**Internal packages used:** `internal/config`

### Purpose

Validate the entire aide configuration without launching anything. Catch structural errors, broken references, missing binaries, and bad YAML. Every error must include an actionable fix suggestion. Validation also runs implicitly on every `aide` launch (but `aide validate` runs it standalone and reports all errors, not just the first).

### Validation Rules

| # | Check | Error message template | Severity |
|---|-------|----------------------|----------|
| V1 | YAML parse error | `config.yaml: line {n}: {yaml_error}. Check indentation and quoting.` | error |
| V2 | Unknown top-level key | `config.yaml: unknown key "{key}". Valid keys: agents, contexts, mcp, default_context, agent, env, secret, mcp_servers, sandbox. Did you mean "{suggestion}"?` | error |
| V3 | Context references unknown agent | `contexts.{ctx}.agent references "{agent}" but no agent named "{agent}" is defined in agents. Available agents: {list}. Add it under agents: or fix the name.` | error |
| V4 | Agent binary not on PATH | `agents.{name}.binary "{binary}" not found on PATH. Install it or update the binary path.` | warning |
| V5 | Context references unknown MCP server | `contexts.{ctx}.mcp_servers references "{server}" but no server named "{server}" is defined in mcp.servers. Available servers: {list}.` | error |
| V6 | Secrets file does not exist | `contexts.{ctx}.secret "{file}" not found at {full_path}. Create it with: aide secrets create {name}` | warning |
| V7 | Template references unknown secret key | `contexts.{ctx}.env.{var}: template references ".secrets.{key}" — this will be validated at runtime when secrets are decrypted.` | info |

> **Nice-to-have (V7 enhancement):** If the secrets file is available and decryptable (e.g., the user's age key can open it), validation could optionally decrypt it and check that the referenced keys actually exist, upgrading the message from info to a warning or error for missing keys. This is not required for the initial implementation.
| V8 | Duplicate context match patterns | `contexts.{ctx1} and contexts.{ctx2} have identical match rule: {rule}. The first defined context will win. Consider making patterns more specific.` | warning |
| V9 | default_context references unknown context | `default_context "{name}" is not defined in contexts. Available contexts: {list}.` | error |
| V10 | MCP server override references server not in mcp_servers list | `contexts.{ctx}.mcp_server_overrides.{server}: overrides server "{server}" but it is not listed in mcp_servers. Add "{server}" to mcp_servers or remove the override.` | error |
| V11 | Empty contexts/agents maps | `agents: is defined but empty. Add at least one agent, e.g.: agents:\n  claude:\n    binary: claude` | warning |
| V12 | .aide.yaml references unknown agent | `.aide.yaml at {path}: agent "{agent}" is not defined in global config agents. Available agents: {list}.` | error |
| V13 | .aide.yaml references unknown MCP server | `.aide.yaml at {path}: mcp_servers references "{server}" not defined in global mcp.servers. Available: {list}.` | error |
| V14 | Sandbox path uses unknown template var | `contexts.{ctx}.sandbox.writable: template variable "{var}" is not a recognized variable. Available: .project_root, .runtime_dir` | error |

### Levenshtein Distance for Suggestions

For V2, V3, V5, V9 -- compute edit distance against known valid values and suggest the closest match if distance <= 3. Use a simple Levenshtein implementation (no external dependency needed).

**File:** `internal/config/levenshtein.go`

```go
// Levenshtein computes the edit distance between two strings.
func Levenshtein(a, b string) int

// Suggest returns the closest match from candidates if within maxDistance.
// Returns empty string if no close match found.
func Suggest(input string, candidates []string, maxDistance int) string
```

### Output Format

```
$ aide validate
Validating ~/.config/aide/config.yaml ...

ERROR  contexts.work.agent references "cladue" but no agent named "cladue"
       is defined in agents. Available agents: claude, gemini, codex.
       Did you mean "claude"?

WARN   agents.codex.binary "codex" not found on PATH.
       Install it or update the binary path.

WARN   contexts.work.secret "work" not found at
       ~/.config/aide/secrets/work.
       Create it with: aide secrets create work

INFO   contexts.work.env.AWS_PROFILE: template references ".secrets.aws_profile"
       — this will be validated at runtime when secrets are decrypted.

Validation complete: 1 error, 2 warnings, 1 info.
```

Exit code: 0 if no errors, 1 if any errors. Warnings do not cause non-zero exit.

### Flags

| Flag | Description |
|------|-------------|
| `--project-file <path>` | Also validate a specific `.aide.yaml` file |
| `--json` | Output validation results as JSON |
| `--strict` | Treat warnings as errors (exit 1 on warnings) |

### Implementation

1. `config.Validate(cfg *Config) []ValidationResult` -- returns all issues found (does not short-circuit).
2. Each `ValidationResult` has `Severity` (error/warning/info), `Code` (V1-V14), `Message`, `Suggestion`.
3. The validate command loads config, calls `config.Validate()`, formats output, and sets exit code.
4. The main launch path (`aide` with no subcommand) calls the same `config.Validate()` and aborts on errors.

### Required Types

```go
// internal/config/validate.go

type Severity int

const (
    SeverityInfo Severity = iota
    SeverityWarning
    SeverityError
)

type ValidationResult struct {
    Severity   Severity
    Code       string // "V1", "V2", etc.
    Message    string
    Suggestion string // actionable fix
    Location   string // e.g., "contexts.work.agent"
}

// Validate runs all validation checks and returns all issues found.
func Validate(cfg *Config, projectOverride *ProjectConfig) []ValidationResult
```

## Task 25: `aide init` — Create .aide.yaml in Current Project

**File:** `cmd/aide/init.go`
**Internal packages used:** `internal/config`, `internal/context`

### Purpose

Create a `.aide.yaml` file in the current project root (git root, or cwd if not a git repo). Pre-fills with sensible defaults based on the currently resolved context. This lets users customize a single project without editing the global config.

### Behavior

1. Determine project root via `context.DetectProjectRoot()` (git root or cwd).
2. Check if `.aide.yaml` already exists at project root. If so, abort with: `".aide.yaml already exists at {path}. Edit it directly or remove it first."`
3. Resolve the current context via the normal resolution pipeline (without the project override, since it doesn't exist yet).
4. Generate a `.aide.yaml` with commented-out fields pre-filled from the resolved context.
5. Write the file and print its path.

### Generated File

```yaml
# .aide.yaml — project-level aide overrides
# Uncomment and modify fields to override your global context settings.
# This file takes precedence over context matching from ~/.config/aide/config.yaml.

# Override the agent for this project:
# agent: claude

# Override MCP servers for this project:
# mcp_servers:
#   - git
#   - context7

# Additional environment variables for this project:
# env:
#   CUSTOM_VAR: "value"
```

When aide can resolve a context, include context-aware comments:

```yaml
# .aide.yaml — project-level aide overrides
# Current context: personal (matched via remote "github.com/jskswamy/*")
# Uncomment and modify fields to override settings from the personal context.

# agent: claude          # currently: claude
# mcp_servers:           # currently: [git, context7, serena, things]
#   - git
#   - context7
#   - serena
#   - things
# env:
#   CUSTOM_VAR: "value"
```

### Flags

| Flag | Description |
|------|-------------|
| `--force` | Overwrite existing `.aide.yaml` |
| `--agent <name>` | Pre-fill agent field (uncommented) |
| `--bare` | Generate minimal file without comments |

### Implementation

1. `context.DetectProjectRoot()` -- walks up from cwd looking for `.git/`, falls back to cwd.
2. Check file existence; respect `--force`.
3. If global config exists, resolve context and use it for comment hints.
4. If no global config, generate a generic template.
5. Write file. Print: `Created .aide.yaml at {path}`.

## Task 26: `aide setup` — Interactive Wizard

**File:** `cmd/aide/setup.go`
**Internal packages used:** `internal/config`, `internal/secrets`

### Purpose

Interactive first-time setup wizard. Detects existing infrastructure (age keys, agent binaries), guides the user through creating a minimal working configuration. Designed so a new user can go from zero to working `aide` in under 2 minutes.

> **Note:** The wizard should also detect existing `.aide.yaml` files in the current working directory and mention them to the user (e.g., "Found project-level `.aide.yaml` in this directory — your global config will be merged with these project overrides at runtime.").

### Wizard Flow

```
$ aide setup

aide setup — first-time configuration wizard

Step 1/4: Age Key Detection
  Checking for age keys...
  [x] Found age key file at ~/.config/sops/age/keys.txt
  [ ] No YubiKey with age-plugin-yubikey detected
  [ ] SOPS_AGE_KEY env var not set

  Your age public key: age1abc123...
  This will be used to encrypt secrets files.

  > Continue with this key? [Y/n]

Step 2/4: Agent Detection
  Scanning PATH for known agent binaries...
  [x] claude   /usr/local/bin/claude
  [x] codex    /usr/local/bin/codex
  [ ] gemini   (not found)

  > Which agent should be the default? [claude]

Step 3/4: Configuration
  Creating ~/.config/aide/config.yaml ...

  > Do you want multi-context support (work/personal)? [y/N]

  (If N — single context mode)
  Generated minimal config:
    agent: claude
    # Add env vars and secrets later with `aide secrets create`

  (If Y — multi-context mode)
  > Enter git remote pattern for work context (e.g., github.com/your-org/*): github.com/work-org/*
  > Enter git remote pattern for personal context (e.g., github.com/yourname/*): github.com/jskswamy/*

  Generated multi-context config with work + personal contexts.

Step 4/4: Secrets (optional)
  > Do you want to create an encrypted secrets file now? [y/N]

  (If Y)
  > Name for secrets file [personal]: personal
  Opening $EDITOR to enter secrets...
  (Uses aide secrets create flow)

  (If N)
  Skipping secrets setup. You can add them later with `aide secrets create`.

Setup complete!
  Config:  ~/.config/aide/config.yaml
  Run `aide` to launch claude in this project.
  Run `aide which` to see what context would be used.
  Run `aide validate` to check your config.
```

### Age Key Detection Details

Check in order (matching the discovery order from DESIGN.md):

1. **YubiKey:** Run `age-plugin-yubikey --identity` and check exit code. If available, extract public key.
2. **`SOPS_AGE_KEY` env var:** Check `os.Getenv("SOPS_AGE_KEY")`. If set, extract public key from it.
3. **`SOPS_AGE_KEY_FILE`:** Check env var, then read the file.
4. **Default key file:** Read `$XDG_CONFIG_HOME/sops/age/keys.txt`.

If no key found anywhere:

```
  No age key found. You have three options:
  1. Generate a new age key (recommended)
  2. Set up YubiKey with age-plugin-yubikey
  3. Skip (you can use aide without encrypted secrets)

  > Choose [1/2/3]:
```

Option 1 generates a key:

```go
// Use filippo.io/age library to generate a key
identity, err := age.GenerateX25519Identity()
// Write to $XDG_CONFIG_HOME/sops/age/keys.txt (standard location)
```

Print the public key and remind the user to back it up.

### Agent Detection Details

Scan for known agent binaries using `exec.LookPath()`:

```go
var knownAgents = []string{"claude", "gemini", "codex"}
```

For each found binary, resolve its absolute path. If zero found, print install guidance and offer to continue without an agent (config-only setup).

### Non-Interactive Mode

```
aide setup --non-interactive --agent claude
```

For CI/scripted environments. Uses all defaults, requires `--agent` flag. Creates minimal config. Skips secrets. Exits 0 on success.

### Flags

| Flag | Description |
|------|-------------|
| `--non-interactive` | Skip all prompts, use defaults |
| `--agent <name>` | Pre-select default agent (required with `--non-interactive`) |
| `--skip-secrets` | Skip the secrets setup step |
| `--force` | Overwrite existing config.yaml |

### Implementation

**File:** `cmd/aide/setup.go` — cobra command and wizard orchestration.
**File:** `internal/setup/wizard.go` — testable wizard logic with injected I/O.

The wizard logic should accept an `io.Reader` and `io.Writer` for input/output so it can be tested without actual stdin/stdout:

```go

// internal/verbose/logger.go

package verbose

import (
    "fmt"
    "io"
    "os"
)

var (
    enabled bool
    writer  io.Writer = os.Stderr
)

// Enable turns on verbose logging.
func Enable() { enabled = true }

// SetWriter overrides the output writer (for testing).
func SetWriter(w io.Writer) { writer = w }

// Log prints a verbose message if verbose mode is enabled.
func Log(format string, args ...interface{}) {
    if enabled {
        fmt.Fprintf(writer, "[aide] "+format+"\n", args...)
    }
}
```

The `-v` / `--verbose` flag is registered as a persistent flag on the root cobra command so it works with all subcommands:

```go
// cmd/aide/root.go
rootCmd.PersistentFlags().BoolVarP(&verboseFlag, "verbose", "v", false, "Show resolution steps")
```

```go
// internal/verbose/redact.go

// Redact shows first 6 chars + "****..." for secret values.
// If value is shorter than 10 chars, show "****" entirely.
func Redact(value string) string {
    if len(value) < 10 {

    }
    return value[:6] + "****..."
}
```

Non-secret env values (literals, not from template) are shown in full.

```go
func newCompletionCmd() *cobra.Command {
    cmd := &cobra.Command{
        Use:   "completion [bash|zsh|fish]",
        Short: "Generate shell completion scripts",
        Long:  completionLongHelp,
        ValidArgs: []string{"bash", "zsh", "fish"},
        Args:  cobra.ExactValidArgs(1),
        RunE: func(cmd *cobra.Command, args []string) error {
            switch args[0] {
            case "bash":
                return cmd.Root().GenBashCompletionV2(cmd.OutOrStdout(), true)
            case "zsh":
                return cmd.Root().GenZshCompletion(cmd.OutOrStdout())
            case "fish":

        },
    }
```

```go
// cmd/aide/secrets.go — on edit/rotate subcommands
ValidArgsFunction: func(cmd *cobra.Command, args []string, toComplete string) ([]string, cobra.ShellCompDirective) {
    if len(args) > 0 {
        return nil, cobra.ShellCompDirectiveNoFileComp
    }
    secretsDir := filepath.Join(config.Dir(), "secrets")
    entries, err := os.ReadDir(secretsDir)
    if err != nil {
        return nil, cobra.ShellCompDirectiveNoFileComp
    }
    var names []string
    for _, e := range entries {
        name := e.Name()
        if strings.HasSuffix(name, ".enc.yaml") {
            names = append(names, strings.TrimSuffix(name, ".enc.yaml"))
        }
    }
    return names, cobra.ShellCompDirectiveNoFileComp
}
```

## Task 30: First-Run Guidance

**File:** `cmd/aide/firstrun.go`
**Internal packages used:** `internal/config`

### Purpose

When a user runs `aide` for the very first time (no config, no sentinel file), show a one-time hint about `aide setup`. On subsequent runs, do not show it again -- even if there's still no config. This prevents nagging while ensuring new users get oriented.

### Behavior

On first ever run with no config:

```
$ aide
Welcome to aide! It looks like this is your first time running aide.

Tip: Run `aide setup` to create your configuration interactively.
     Or just keep going — aide will auto-detect agents on your PATH.

(This message will not appear again.)

Detected claude on PATH. Launching...
```

The hint is shown BEFORE the zero-config passthrough logic runs. The agent still launches -- the hint is informational and does not block.

### Sentinel File

**Location:** `$XDG_CONFIG_HOME/aide/.first-run-shown`

This is a zero-byte file. Its existence means the hint has been shown. It is NOT inside the `secrets/` directory and is safe to git-track (though it adds no value to track).

### Logic

```go
// internal/config/firstrun.go

// SentinelPath returns the path to the first-run sentinel file.
func SentinelPath() string {
    return filepath.Join(Dir(), ".first-run-shown")
}

// IsFirstRun returns true if the sentinel file does not exist.
func IsFirstRun() bool {
    _, err := os.Stat(SentinelPath())
    return os.IsNotExist(err)
}

// MarkFirstRunShown creates the sentinel file.
func MarkFirstRunShown() error {
    // Ensure config dir exists
    if err := os.MkdirAll(Dir(), 0700); err != nil {
        return err
    }
    f, err := os.Create(SentinelPath())
    if err != nil {
        return err
    }
    return f.Close()
}
```

### Integration Point

In the root command's `RunE` (the default launch path), before context resolution:

```go
// cmd/aide/root.go — in the main launch RunE

if config.IsFirstRun() && !config.Exists() {
    fmt.Fprintln(os.Stderr, firstRunMessage)
    config.MarkFirstRunShown()
}
```

The check requires BOTH conditions:
- Sentinel does not exist (`IsFirstRun()`)
- No config.yaml exists (`!config.Exists()`)

This means: if a user creates config.yaml manually before running aide, they never see the hint (they don't need it). If they delete config.yaml later, they still don't see it (sentinel exists). The hint targets only true first-time users.

### First-Run Message

```go
const firstRunMessage = `Welcome to aide! It looks like this is your first time running aide.

Tip: Run "aide setup" to create your configuration interactively.
     Or just keep going — aide will auto-detect agents on your PATH.

(This message will not appear again.)`
```

### `aide setup` Also Creates Sentinel

When `aide setup` completes successfully, it should also call `config.MarkFirstRunShown()` so users who run `aide setup` first never see the first-run hint.

## Cross-Cutting Concerns

### Error Output Convention

All informational and error output goes to **stderr**. Only structured output (JSON, completion scripts) goes to stdout. This ensures aide's output never pollutes agent stdin/stdout.

### Testing Strategy

All commands use testable patterns:
- Commands accept `*cobra.Command` with overridable `OutOrStdout()` / `OutOrStderr()`.
- Business logic lives in `internal/` packages with injected dependencies (no global state).
- Tests use `t.TempDir()` for config/secrets directories.
- Tests override `exec.LookPath` via function injection (not direct calls).
- No tests depend on real filesystem state outside temp dirs.

### JSON Output

Commands that support `--json` use `encoding/json` with `MarshalIndent` (2-space indent) for human-readable JSON output. JSON output goes to stdout. When `--json` is used, no non-JSON output is printed to stdout (verbose and errors still go to stderr).
